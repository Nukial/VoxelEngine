# Known Issues and Research Problems

This document records the current technical problems, limitations, and research questions in VoxelEngine.

VoxelEngine is an experimental GPU voxel simulation and ray-marched rendering prototype. Many of the items below are not simple bugs; they are architectural problems that appear when a voxel world is stored and simulated primarily on the GPU while Unity remains the host engine.

## 1. Unity pipeline mismatch

Unity's default strengths are meshes, GameObjects, renderers, batching, culling, colliders, lighting, prefabs, and editor tooling.

VoxelEngine takes a different route:

- voxel state is stored in GPU buffers,
- simulation runs in compute shaders,
- rendering is done through ray marching,
- Unity's normal mesh pipeline is partially bypassed.

This makes the project useful for research, but it means many built-in Unity systems do not automatically work with the voxel world.

## 2. CPU/GPU synchronization

A game often needs immediate CPU-side answers:

- what voxel did the player click?
- what material is at this position?
- can an entity move through this cell?
- should AI pathfinding avoid this region?
- what changed after the latest simulation step?

When the authoritative world state is on the GPU, reading the data back to the CPU can add latency and performance overhead.

Possible research directions:

- async GPU readback cache,
- partial region readback,
- gameplay proxy grid,
- hybrid CPU/GPU ownership,
- event buffers generated by compute shaders,
- GPU-side picking for interaction.

## 3. Physics and collision are incomplete

Unity Physics does not understand a custom voxel field stored in `GraphicsBuffer`.

Possible approaches:

- generate simplified colliders from voxel regions,
- keep a CPU-side collision proxy,
- use signed distance fields or occupancy grids,
- use chunk-level broadphase and voxel-level narrowphase,
- restrict physical interaction to selected regions.

## 4. Large world streaming is not implemented

The current prototype is best suited for bounded volumes and visual simulations.

A game-scale voxel world would need:

- chunk paging,
- region streaming,
- serialization,
- save/load,
- GPU memory budgeting,
- partial simulation updates,
- inactive region sleeping,
- cross-chunk simulation boundaries.

## 5. Editing many voxels needs a better update path

Small edits are acceptable for experimentation, but large edits should not rely on many small CPU-to-GPU writes.

Possible future solution:

- upload compact edit command buffers,
- process edits directly in compute shaders,
- batch sphere/box/brush operations,
- rebuild dirty brick-map regions only,
- avoid full-buffer synchronization.

## 6. Ray-marched rendering has trade-offs

Ray marching can render a voxel volume without converting every visible face into mesh geometry, but it has its own costs:

- ray step count can become expensive,
- shadows add more traversal cost,
- transparency requires extra handling,
- edge cases near the volume boundary can be expensive,
- depth integration with normal Unity objects needs care,
- visual quality depends heavily on shader tuning.

## 7. Brick-map acceleration is useful but limited

The brick map helps skip empty regions, but it is not a complete spatial acceleration system.

Open questions:

- should brick maps become hierarchical?
- should dirty bricks be updated incrementally?
- how large should each brick be?
- how should sparse regions be represented?
- how should dynamic materials affect brick-map updates?

## 8. Simulation stability is still experimental

The current compute shader simulation includes powder, liquid, gas, heat, light, and material state changes. This is visually interesting, but still experimental.

Known risks:

- flickering from multi-pass light propagation,
- unintuitive material behavior,
- race-condition-like artifacts,
- unstable tuning across frame rates,
- heavy cost at high world sizes,
- platform-specific compute shader behavior.

## 9. Gameplay meaning is not solved by simulation alone

A system can simulate sand, water, lava, steam, fire, heat, and light, but that does not automatically make a good game.

A real game would still need:

- goals,
- progression,
- player tools,
- AI,
- resources,
- UI,
- save/load,
- authored content,
- level design,
- performance budgets,
- clear interaction rules.

## 10. Recommended research direction

The most practical direction is likely hybrid:

- stable terrain: chunk mesh or another conventional representation,
- special dynamic regions: GPU voxel simulation,
- visual effects: ray-marched voxel volume,
- gameplay data: CPU-accessible proxy or selective readback,
- Unity: editor, input, UI, audio, scene hosting, and build pipeline.

This keeps the GPU research value while avoiding the trap of forcing the entire game to depend on a hard-to-query volume buffer.
